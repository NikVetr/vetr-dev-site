# Color Palette Optimization Summary

This project is a browser-based tool for extending discrete color palettes with color-vision-deficiency-aware optimization. The app is served as static HTML, CSS, and JavaScript and runs the optimizer entirely in the browser.

## Architecture

- `index.html`, `style.css`, and `main.js` define the application shell, controls, event flow, optimization orchestration, history, and result selection.
- `core/` contains color-space conversion, CVD simulation, perceptual distance metrics, aggregation means, random seeding, and resolvability utilities.
- `optimizer/` contains constraint/bounds construction, the objective function, Nelder-Mead search, and restart initialization. Gamut-aware optimization projects candidate colors into the selected display gamut when optimization clipping is enabled.
- `ui/` contains rendering and interaction modules for the four-panel color wheel, optimization path mini-view, status graph, swatches, results, image-derived color picking, and direct manipulation of constraints and palette points.
- `tests/` contains focused Node tests for color math, objectives, starts, and metrics plus Playwright specs for browser screenshots and visual smoke coverage.

## Key Behavior

The optimizer works in the selected raw optimization color space and can evaluate palettes under normal vision plus deutan, protan, and tritan simulations. Visual gamut clipping is enabled by default as a rendering setting; optimization gamut clipping is a separate optimization setting. Both use the selected gamut preset, and clipped visualizations share the same projected gamut boundary and out-of-gamut overlay treatment. Clipped optimization retains optimizer-space coordinates for diagnostics and constraint enforcement, while clipped visualizations plot gamut-projected display positions for palette points and path traces; main-panel input and output markers are drawn as un-clipped glyphs above the gamut and constraint masks. The main color wheels and optimization paths view draw the full projected gamut as a solid outline and any active hard contiguous hidden-axis constraint-aware gamut as a dashed outline, so defaults such as OKLab L can be compared against the full display gamut. Visible-axis constraints are drawn as guides and do not change the hidden-axis gamut slice; for example, adding OKLab A/B constraints does not expand or reshape the L-constrained gamut boundary. Hand-drawn custom constraints are recorded from raw wheel coordinates, even when the drag starts in an out-of-gamut part of the canvas. Slider-based guides and hand-drawn custom rectangles are clipped to the active projected gamut silhouette, and custom hidden-axis hard windows draw their own dashed constraint-aware gamut silhouettes so rectangular raw constraints are shown with the hidden-channel feasible intersection. The part of the full gamut outside the active constraint-aware region is faded in the main panels using a union mask, so overlapping custom constraint regions stay rich rather than toggling between included and excluded. Soft constraints are rendered as penalty fades on the color wheels and channel bars, not as feasible-domain outlines in the paths panel. HSL uses its native `[0, 100]` saturation/lightness domain for the full gamut, while Luv boundary sampling avoids the `L=0` singularity so arbitrary nonzero `u/v` coordinates at black do not inflate the drawn gamut. The visible path-panel gamut boundary is drawn from the exact projected boundary, and path strokes are clipped to that boundary when visual gamut clipping is enabled.

Constraint bounds are derived from input colors, custom constraint points, raw overrides, or empty-palette defaults. Those bounds are shared by the UI and optimizer so default hard constraints are visible and active before any input colors are added. Default lightness constraints are calibrated heuristically by colorspace to approximate OKLab `L = 65%`: OKLab/OKLCh use `65%`, Lab/LCh/Luv use `76.9%`, HSL uses `73.1%`, and JzAzBz uses `71%`. Custom constraints remain literal even when aesthetic modes are enabled, and hard custom/discontiguous windows enforce feasibility without adding a center-seeking penalty for colors already inside an allowed window.

Result selections are copied into editable UI state rather than aliasing ranked optimizer results. When output colors are edited through the wheel or channel bars, raw output coordinates are kept in a single consistent color space before rendering or further edits.

Optimization-path rendering stores a sampled best-point trajectory from each Nelder-Mead restart instead of only connecting each restart's initial and final palette. The paths panel draws those sampled trajectories with a blue-to-red time gradient, falling back to start-to-end strokes only when older result objects do not contain trajectory samples. Users can choose the number of sampled trajectory steps per restart, defaulting to 48 and capped by the configured Nelder-Mead iteration count. Trajectory strokes and current markers are clipped to the visible hard constraint domain as well as the active projected gamut, and disjoint custom-window paths break when a sample switches between allowed windows instead of drawing straight connectors through infeasible space. Best-result stars are drawn in a final top overlay after confirming that their display coordinates are inside the visible hard-constraint domain; already in-gamut optimizer display coordinates are preserved so a second constraint-blind projection cannot hide those stars. The paths panel also draws visible hard custom/discontiguous constraint windows as dashed guides so hand-drawn constraints are visible alongside slider-based constraints, including before an optimization run starts.

Image inputs can be added by upload, paste, or drag and drop anywhere on the page. The page-wide drop affordance explains that image processing happens locally in the browser. The image picker uses quantized color peaks with OKLab-distance spacing to seed distinct candidate input points, supports adaptive auto cluster counts or an explicit user-selected count with a re-cluster action, and keeps clustering diagnostics in a collapsed info panel. Users can add, drag, and delete image-sampled points in a focused modal with a pixel magnifier. Image-derived swatches include normal hex labels plus deutan, protan, and tritan simulation bands. Image-derived colors are written into the normal hex textarea as additions and tracked so reopening the image picker updates or removes the corresponding hex entries instead of appending duplicates. Built-in starter palettes include ColorBrewer-style sets, scientific palettes, the HTML 4.01 named color set, and common categorical palettes.
