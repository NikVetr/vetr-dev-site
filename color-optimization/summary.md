# Color Palette Optimization Summary

This project is a browser-based tool for extending discrete color palettes with color-vision-deficiency-aware optimization. The app is served as static HTML, CSS, and JavaScript and runs the optimizer entirely in the browser.

## Architecture

- `index.html`, `style.css`, and `main.js` define the application shell, controls, event flow, optimization orchestration, history, and result selection.
- `core/` contains color-space conversion, CVD simulation, perceptual distance metrics, aggregation means, random seeding, and resolvability utilities.
- `optimizer/` contains constraint/bounds construction, the objective function, Nelder-Mead search, and restart initialization. Gamut-aware optimization projects candidate colors into the selected display gamut when optimization clipping is enabled.
- `ui/` contains rendering and interaction modules for the four-panel color wheel, optimization path mini-view, status graph, swatches, results, image-derived color picking, and direct manipulation of constraints and palette points.
- `tests/` contains focused Node tests for color math, objectives, starts, and metrics plus Playwright specs for browser screenshots and visual smoke coverage.

## Key Behavior

The optimizer works in the selected raw optimization color space and can evaluate palettes under normal vision plus deutan, protan, and tritan simulations. Visual gamut clipping is enabled by default as a rendering setting; optimization gamut clipping is a separate optimization setting. Both use the selected gamut preset, and clipped visualizations share the same projected gamut boundary and out-of-gamut overlay treatment. Clipped optimization retains optimizer-space coordinates for diagnostics and constraint enforcement, while distance scoring uses the projected output coordinates that are returned as hex colors. Clipped visualizations plot gamut-projected display positions for palette points and path traces; main-panel input and output markers are drawn as un-clipped glyphs above the gamut and constraint masks. The main color wheels and optimization paths view draw the full projected gamut as a solid outline and any active hard contiguous hidden-axis constraint-aware gamut as a dashed outline, so defaults such as OKLab L can be compared against the full display gamut. Visible-axis constraints are drawn as guides and do not change the hidden-axis gamut slice; for example, adding OKLab A/B constraints does not expand or reshape the L-constrained gamut boundary. Hand-drawn custom constraints are recorded from raw wheel coordinates, even when the drag starts in an out-of-gamut part of the canvas. Slider-based guides and hand-drawn custom rectangles are clipped to the active projected gamut silhouette, and custom hidden-axis hard windows draw their own dashed constraint-aware gamut silhouettes so rectangular raw constraints are shown with the hidden-channel feasible intersection. The part of the full gamut outside the active constraint-aware region is faded in the main panels using a union mask, so overlapping custom constraint regions stay rich rather than toggling between included and excluded. Soft constraints are rendered as penalty fades on the color wheels and channel bars, not as feasible-domain outlines in the paths panel. HSL uses its native `[0, 100]` saturation/lightness domain for the full gamut, while Luv boundary sampling avoids the `L=0` singularity so arbitrary nonzero `u/v` coordinates at black do not inflate the drawn gamut. The visible path-panel gamut boundary is drawn from the exact projected boundary, and path strokes are clipped to that boundary when visual gamut clipping is enabled.

Constraint bounds are derived from input colors, custom constraint points, raw overrides, or empty-palette defaults. Those bounds are shared by the UI and optimizer so default hard constraints are visible and active before any input colors are added. Default lightness constraints are calibrated heuristically by colorspace to approximate OKLab `L = 65%`: OKLab/OKLCh use `65%`, Lab/LCh/Luv use `76.9%`, HSL uses `73.1%`, and JzAzBz uses `71%`. Custom constraints remain literal even when aesthetic modes are enabled, and hard custom/discontiguous windows enforce feasibility without adding a center-seeking penalty for colors already inside an allowed window. Dragging a new custom constraint while another topology is active switches to `Custom` and starts a clean custom constraint set from the drawn region instead of inheriting the prior palette-derived windows. The aesthetic-centers dropdown is disabled with a tooltip whenever the current topology, colorspace, or input state means the setting would not change the computed constraints.

Individuated constraints are available under discontiguous topology and maintain per-input widths and hard/soft modes for the three active channel slots. The global channel sliders are labeled as global constraints, while the individuated table controls input-specific constraints. Each table row has its own hard/soft toggle; newly enabled tweak rows default to soft mode with 85% widths across all channels, while ordinary rows default to the global channel mode unless changed. The table also includes a rainbow `All` row that can set every row's mode, set a shared width for a channel, or toggle tweaking for every input at once. Per-input sliders keep the table compact by showing their numeric percentage in a hover/focus bubble near the slider handle, and double-clicking a slider opens exact keyboard entry for that percentage. When this mode is active, the visualization colorspace is synchronized to the optimization colorspace so the displayed per-input windows match the constraints used by the optimizer. Input colors can be marked for tweaking from the input swatch controls, the individuated-constraints table, or the input-panel `Tweak All` / `Untweak All` button. Tweaked inputs become optimized rows, do not count against `Colors to add`, and are constrained to their own source input's point window rather than the global discontiguous union. Tweak rows use the same hard/soft constraint machinery as other individuated constraints, applied only to the corresponding tweaked output color; there is no separate tweak-divergence weight. Hard and soft tweak windows are excluded from the candidate windows used by ordinary added outputs, so a local tweak constraint cannot become a global hard region or soft penalty for new colors. Auto-enabled tweak constraints remain layered with global constraints, but the global discontiguous layer is rebuilt without the source rows currently being tweaked; this keeps non-tweak global constraints visible and active without letting a tweak-local window constrain new colors. Global constraints are skipped for a specific tweaked output only if hard global and hard tweak windows for that row have no compatible overlap. Manually enabled individuated constraints replace the global constraint set as the active source for optimization starts, hard clamping, penalties, gamut projection, channel bars, main colorspace panels, and optimization paths. Tweak rows keep their input identities rather than sharing the anonymous lightness-sorted parameterization used for newly added colors, and the first optimization restart is anchored at the source tweak colors so a previous optimum is available as an available candidate. Tweaked inputs remain visible as dashed reference markers, tweaked outputs use dashed output styling, and solid swatch connectors link each tweaked input to its optimized output in the main panels; the optimization-path view draws the corresponding optimized points as normal top-layer stars. Added outputs fill untweaked swatch rows above and below pinned tweaked outputs instead of being forced below the last tweaked row. Global constraints are the base visible constraints in the main colorspace canvases, channel bars, and optimization-path view unless custom or manually individuated constraints are the active source; per-tweak constraint windows are hidden by default and are drawn as an additional hover/focus layer when the corresponding tweak marker, swatch badge, or individuated slider is targeted. Soft tweak previews show the soft penalty fade without the normal teal guide outline; hard tweak previews still show hard boundaries. Collapsed 100% hard tweak windows render as a point-local dashed ring so exact-point constraints remain visible. When tweak constraints are auto-enabled, inactive or toggled-off input rows are reset to the current global defaults instead of persisting as ordinary per-input constraint windows; non-tweaked rows are also filtered out of the auto-enabled local constraint layer so they do not silently constrain new outputs. Manually enabled individuated constraints are preserved as normal row-specific constraints. Main-panel point glyphs use the same source hex colors as the swatch badges and channel-bar dots, while their positions may still use gamut-projected display coordinates when clipping is enabled.

Result selections are copied into editable UI state rather than aliasing ranked optimizer results. When output colors are edited through the wheel or channel bars, raw output coordinates are kept in a single consistent color space before rendering or further edits. Resolvability panels use the effective palette that the objective compares: fixed inputs plus outputs, with generated tweak outputs replacing their source inputs so a source color is not compared against its own tweak. When generated tweak outputs exist, an `untweak` checkbox in each resolvability panel can instead show the source input colors and omit their corresponding tweak outputs for before/after diagnostics. In the color-wheel constraint overlays, hard-excluded regions use the maximum soft-constraint fade whenever hard and soft constraints are mixed in the displayed channels or an active soft tweak preview is shown, so inaccessible regions never appear as reachable as accessible-but-penalized regions. Soft constraint drag guides are drawn as solid semi-transparent teal handles with a white underlay, while hard constraint boundaries retain the dashed black/white style. Wheel pointer interactions convert browser CSS pixels into the canvas logical coordinate system before hit-testing or dragging. Slider-based constraint edge drags store explicit channel bounds for the active colorspace, including wrapped hue intervals, while the visible sliders show the resulting interval width and return to palette-derived bounds when edited directly. Constraint edge drags snapshot the starting bounds on pointer-down, update only on movement, and redraw immediately instead of using the slower constraint-edit throttle. Custom constraints being drawn on channel bars render live preview boundaries and a temporary constraint marker before mouseup, matching the existing wheel preview behavior. Circular color-space legends stack `constraint` above `full gamut`; rectangular color spaces retain the horizontal legend, and discontiguous hidden hard-window gamut slices are included in the dashed constraint legend.

Optimization-path rendering stores a sampled best-point trajectory from each Nelder-Mead restart instead of only connecting each restart's initial and final palette. The paths panel draws those sampled trajectories with a blue-to-red time gradient, falling back to start-to-end strokes only when older result objects do not contain trajectory samples. Users can choose the number of sampled trajectory steps per restart, defaulting to 48 and capped by the configured Nelder-Mead iteration count. While optimization is running, the primary action becomes a red `STOP` button; stopping cancels at the next optimizer checkpoint and keeps the best result from completed restarts. Trajectory strokes and current markers are clipped to the visible hard constraint domain as well as the active projected gamut, and disjoint custom-window paths break when a sample switches between allowed windows instead of drawing straight connectors through infeasible space. Best-result stars are drawn in a final top overlay without clipping; if a best point falls outside the visible hard-constraint domain it is still drawn with a red violation marker so constraint inconsistencies are visible instead of silently hidden. Already in-gamut optimizer display coordinates are preserved so a second constraint-blind projection cannot hide those stars. The paths panel also draws visible custom/discontiguous constraint windows as dashed guides so hand-drawn and manually individuated constraints are visible alongside slider-based constraints, including before an optimization run starts.

Image inputs can be added by upload, paste, or drag and drop anywhere on the page. The page-wide drop affordance explains that image processing happens locally in the browser. The image picker uses quantized color peaks with OKLab-distance spacing to seed distinct candidate input points, supports adaptive auto cluster counts or an explicit user-selected count with a re-cluster action, and keeps clustering diagnostics in a collapsed info panel. Users can add, drag, and delete image-sampled points in a focused modal with a pixel magnifier. Image-derived swatches include normal hex labels plus deutan, protan, and tritan simulation bands. Image-derived colors are written into the normal hex textarea as additions and tracked so reopening the image picker updates or removes the corresponding hex entries instead of appending duplicates. Built-in starter palettes include ColorBrewer-style sets, scientific palettes, the HTML 4.01 named color set, and common categorical palettes.
